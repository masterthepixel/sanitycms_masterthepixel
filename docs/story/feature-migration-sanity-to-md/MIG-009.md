# MIG-009 — Pure Next.js theme cleanup (remove remaining Sanity CMS)

## Epic

Convert `sanitycms_masterthepixel` into a pure Next.js marketing theme with local, file-based content only. Remove Sanity runtime, Sanity Studio assets, and all remaining project dependencies, documentation, and configuration that assume a Sanity CMS backend.

### Objective

- eliminate Sanity CMS as a runtime dependency
- preserve the existing Next.js theme, layout, and content rendering
- use `content/*` as the single source of truth for pages, posts, news, and case studies
- leverage existing Sanity export data (`content-only.ndjson`, asset maps, Studio export artifacts) when validating content coverage
- remove Sanity Studio and any Sanity deployment/install guidance from the repo

### Success criteria

- no runtime `@sanity/*` or `next-sanity` packages are required
- no Sanity runtime imports remain in `src/`
- the site builds successfully with `pnpm run build`
- `npx tsc --noEmit` passes
- no production or local env var is required for Sanity (`NEXT_PUBLIC_SANITY_PROJECT_ID`, `NEXT_PUBLIC_SANITY_DATASET`, `SANITY_API_READ_TOKEN`, `SANITY_CONTENTDEVELOPER_TOKEN`)
- all public routes render from `content/*` file-based sources
- no stale Sanity Studio files, Studio archive assets, or Sanity-specific docs remain in active repo paths
- `content/*` files are sanitized of stale Sanity URLs, branding, and references

## Scope

### In scope

- content migration and verification for all public pages/routes
- cleanup of Sanity Studio folders and archive files
- cleanup of Sanity-related scripts, tests, docs, and migration artifacts
- cleanup of Sanity-specific image/config settings and content references
- removal of stale Sanity env docs, deployment notes, and local secrets
- verification with build, typecheck, and lint

### Out of scope

- redesigning UI or page layouts
- rebuilding Studio editing experience
- creating a new CMS
- changes to content copy beyond migration support and cleanup

## Architectural Edge Cases & Mitigation

To guarantee zero data loss and identical UI rendering during the transition to a pure Next.js application, the following edge cases must be explicitly addressed during execution:

1. **Build-Time Redirects (`next.config.ts`)**
   - **Risk**: The current `next.config.ts` fetches redirects dynamically from the Sanity backend at build time. Deleting Sanity environments will crash the build.
   - **Mitigation**: Rewrite the Next.js `redirects()` function to parse a local JSON map (e.g., `data/redirects.json`) instead of querying the CMS.

2. **Image Cropping & Resizing (Hotspot Data)**
   - **Risk**: Sanity's CDN dynamically crops images via query parameters (e.g., `?w=1200&fit=crop`). Local assets in `/public/assets` are strictly original raw files.
   - **Mitigation**: Ensure UI components utilizing images use standard `next/image` properties (`fill`, `sizes`, `style={{ objectFit: 'cover' }}`) to replicate the visual bounding boxes natively without relying on CDN URL params.

3. **Orphaned Type Definitions (`sanity.types.ts`)**
   - **Risk**: The Next.js frontend is strictly typed using Sanity's code-generated types (`sanity.types.ts`). If this file is indiscriminately deleted, `tsc` will fail globally.
   - **Mitigation**: Retain `sanity.types.ts` as a static, legacy definitions file in the short term. The types will accurately reflect the shape of the data stored in the local `.mdx`/JSON files. Long-term, slowly migrate these to native TypeScript interfaces.

4. **Draft & Preview Mode API Routes**
   - **Risk**: The repository contains API routes (e.g., `src/app/(backend)/api/draft/...`) and libraries (`src/sanity/lib/live.ts`) configured to stream live preview updates from the CMS.
   - **Mitigation**: Scrutinize and remove the backend draft API routes and the Sanity live client entirely to prevent the application from opening streaming connections to a removed CMS.

5. **The Authoring Experience Transition**
   - **Risk**: Editors will lose the Sanity Studio GUI at `/studio`.
   - **Mitigation**: Document the new publishing workflow (writing MDX in VS Code, Git commits, Vercel deployments) in the `README.md` to establish the new standard operating procedure.

## Phases

### Phase 1 — Audit existing Sanity artifacts

- confirm `package.json` contains no frontend Sanity runtime deps
- identify all Sanity references in `src/` and repo docs
- inventory `content/*` files used by `src/lib/content.ts`
- review `content-only.ndjson`, `data/asset-map.json`, and existing Studio export artifacts to confirm local content coverage
- identify any remaining `pageBuilder` JSON or local page fallback content
- identify content files that still contain Sanity URLs, Sanity branding, or Sanity-specific copy
- identify local env files and repo files that still contain Sanity secrets or tokens

### Phase 2 — Content migration and coverage

- verify `content/pages/*.mdx` exists for top-level routes
- verify `content/pages/projects/*.mdx` and `content/pages/services/*.mdx` exist for nested routes
- verify `content/posts/*.mdx`, `content/news/*.mdx`, and `content/case-studies/*.mdx` cover published content
- use existing Sanity export files and asset mappings to ensure all live content and images are migrated
- resolve any `pageBuilder` data that still depends on Sanity IDs or references
- replace JSON `pageBuilder` fallback pages with local MDX or JSON content if needed

### Phase 3 — Sanity cleanup

- archive or remove `studio/masterthepixel`
- remove `archive/studio` if it is legacy only
- remove Sanity migration and asset scripts such as `check-content.js`, `scripts/sanity-to-mdx.js`, `scripts/download-assets.js`, `scripts/extract-redirects.js`, `scripts/fix-asset-assignments.js`, and `scripts/verify-asset-migration.js`
- remove migration-only tests such as `tests/unit/converter.spec.ts`
- remove `src/components/shared/install-demo-button.tsx` Sanity instructions if not needed
- remove or archive Sanity-specific docs in `docs/`
- sanitize `content/*` files for stale `cdn.sanity.io` asset URLs, Sanity branding, and Sanity-specific copy
- remove local environment secrets and docs referencing Sanity tokens

### Phase 4 — Config and deployment cleanup

- remove `cdn.sanity.io` image remotePatterns from `next.config.ts` if no remote Sanity assets are used
- remove Sanity environment variable references from `README.md`, docs, and CI instructions
- remove `sanity.types.ts` and any other legacy Sanity type artifacts if still present and unused
- remove stale GitHub workflow branches or docs that still reference `feature/migration/sanity-to-md`
- ensure Vercel / CI config no longer requires Sanity envs

### Phase 5 — Validation

- run `pnpm install`
- run `pnpm run build`
- run `npx tsc --noEmit`
- run `pnpm run lint`
- optionally add smoke tests for key page render paths

### Must not miss

- remove all Studio folders and archive files that are no longer used
- remove all Sanity migration scripts, migrations-only tests, and helper utilities
- remove all Sanity env references from docs, CI, and local `.env` files
- sanitize `content/*` files for `cdn.sanity.io` URLs, Sanity logo/branding, and Sanity-specific text
- confirm `next.config.ts` no longer depends on `cdn.sanity.io` unless remote assets remain intentionally
- confirm no `Sanity` or `sanity.io` mentions remain in active public-facing site copy or deploy docs

## Deliverables

- `docs/story/feature-migration-sanity-to-md/MIG-009.md` (this epic)
- cleanup PR that removes Sanity Studio assets, scripts, and docs
- `README.md` updated to reflect pure Next.js content sourcing
- validation proof from build and typecheck

## Notes

This repo is already far along in the migration. The main work is cleanup and verification rather than a full rewrite.

Existing Sanity data exports such as `content-only.ndjson` and `data/asset-map.json` should be used as the authoritative migration reference while validating completeness.
